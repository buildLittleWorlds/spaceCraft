Build Guide · Specialized Model Pattern

GLiNER HandyLab

One small model, multiple NLP tasks, all on CPU. This Space shows how a focused model can do more than you'd expect.

View the Space on Hugging Face

What This Space Does

GLiNER HandyLab loads a single lightweight model and uses it for several NLP tasks: named entity recognition, relation extraction, text classification, and structured data extraction. You type text, define what you're looking for (custom entity types, relations, or categories), and the model finds them. It's zero-shot — you don't need to train it for your specific task.

The Architecture

This is the specialized model pattern from Chapter 3. Instead of loading a massive general-purpose model, the Space uses GLiNER — a model designed from the ground up to be lightweight and multi-task while running on CPU.

Why GLiNER works on CPU: GLiNER is a "Generalist and Lightweight" model for entity recognition, published at NAACL 2024. The base version has roughly 200M parameters — well within the CPU-friendly range. It achieves zero-shot capability not by being enormous (like GPT-4) but by being cleverly designed: it learns to match text spans to arbitrary labels at training time, so at inference time you can define any label set you want.

The Code Pattern

Here's the essential structure. This isn't the exact source code — it's the pattern you'd follow to build a similar multi-task NLP Space.

1. Load the model once at startup

from gliner import GLiNER model = GLiNER.from_pretrained("urchade/gliner_medium-v2.1")
The model loads once when the Space starts, not on every request. This is critical for CPU Spaces — model loading takes 10-30 seconds, but inference on a loaded model takes under a second. The medium variant balances accuracy and speed.

2. Define task-specific functions

def extract_entities(text, labels): """NER: find entities matching user-defined types.""" label_list = [l.strip() for l in labels.split(",")] entities = model.predict_entities( text, label_list, threshold=0.5 ) return format_entities(entities) def extract_relations(text, relations): """Relation extraction: find connections.""" # Same model, different method call ... def classify_text(text, categories): """Zero-shot classification.""" # Same model, yet another method ...
The key pattern: one model, multiple functions. Each function calls a different method on the same loaded model. The user defines what to look for at runtime (entity types, relation types, categories) — that's the "zero-shot" part. No retraining needed.

3. Multi-tab Gradio interface

import gradio as gr with gr.Blocks() as demo: gr.Markdown("# GLiNER HandyLab") with gr.Tabs(): with gr.Tab("NER"): text_in = gr.Textbox(label="Text") labels_in = gr.Textbox( label="Entity types (comma-separated)", placeholder="person, organization, location" ) output = gr.Markdown() btn = gr.Button("Extract") btn.click(extract_entities, [text_in, labels_in], output) with gr.Tab("Relations"): # Similar layout, different function ... with gr.Tab("Classification"): # Similar layout, different function ... demo.launch()
gr.Blocks() instead of gr.Interface() gives you full layout control. gr.Tabs() creates the multi-task interface — each tab calls a different function on the same model. This is the standard Gradio pattern for Spaces that do more than one thing.

Why gr.Blocks vs gr.Interface

In Chapter 5, all three starter Spaces use gr.Interface — the simplest way to build a Space. gr.Interface takes one function with inputs and outputs and builds the UI automatically.

gr.Blocks is the next level. It gives you control over layout: tabs, columns, rows, conditional visibility, multiple buttons that trigger different functions. GLiNER HandyLab needs Blocks because it runs multiple tasks from one interface. When you're ready to build something more complex than a single-function tool, gr.Blocks is how you do it.

What You Could Change

  1. Build a domain-specific NER tool. Pre-fill the entity types for a specific domain instead of asking the user to type them. For a legal tool: always extract "plaintiff," "defendant," "court," "date." For a medical tool: "symptom," "medication," "condition." Same model, different framing — that's user-centered design.
  2. Add a confidence threshold slider. The threshold=0.5 in the code controls how confident the model needs to be before reporting an entity. Add a gr.Slider(0, 1, value=0.5) and let users adjust it. Low threshold = find more things, with more false positives. High threshold = find fewer things, with higher precision. That's a parameter sweep.
  3. Chain it with another Space. Use GLiNER as step 1 in a pipeline: extract entities from text, then look up each entity using a free API (Wikipedia, Wikidata, or a dictionary API). The NER output becomes the search input for step 2.

Connection to the Course

  • Session 4 (Classification vs Generation): GLiNER is a classifier — it assigns labels to text spans. Compare its output structure (labels) with a generator's output (new text) to see the fork in action.
  • Session 6 (External Validity): GLiNER is trained primarily on English. Test it on text from other languages or highly technical domains. Where does it break? That's domain shift.

The lesson: You don't need a large model to build a powerful NLP tool. GLiNER is under 500M parameters and handles tasks that would have required separate specialized models five years ago. The zero-shot capability means you can adapt it to any domain without training. For a student building on free CPU, this is the sweet spot: small enough to run, flexible enough to be useful for almost anything involving text.
