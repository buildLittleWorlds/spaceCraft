Chapter 3

GPU vs CPU

Why some Spaces need a GPU, how to tell the difference, and the tricks that make CPU work.

Why GPUs Exist for ML

Machine learning is mostly matrix multiplication. Training a model and running inference both involve multiplying enormous matrices of numbers, millions of times. CPUs do this sequentially — one operation after another. GPUs do it in parallel — thousands of operations at the same time.

For a model with billions of parameters processing a high-resolution image, the difference is dramatic: what takes minutes on CPU can take seconds on GPU. That's why the flashiest Spaces require GPUs.

But not every ML task involves enormous matrices. And not every useful Space even runs an ML model.

Model Size Rules of Thumb

The biggest factor in whether a model runs on CPU is its parameter count:

SizeParamsRAM NeededCPU Viable?
Tiny< 100M< 500 MBYes, fast
Small100M – 500M500 MB – 2 GBYes, reasonable
Medium500M – 1B2 – 4 GBYes, slower
Large1B – 3B4 – 12 GBMaybe, with tricks
Very Large3B+12+ GBUsually not

These are rough numbers. Actual RAM depends on architecture and precision. But as a first approximation: under 1B parameters is usually CPU-friendly. Over 3B usually needs a GPU.

How to Check a Space's Hardware

Every Space on Hugging Face shows its hardware tier:

  1. The Space page header — Look near the title for a badge: "CPU Basic", "T4 small", "A10G", etc.
  2. The README frontmatter — Click "Files" and check for a hardware: field. If it's absent, the Space uses free CPU.
  3. The Settings tab — If you own the Space, Settings shows the hardware selection.

Quick rule: If a Space says "CPU Basic" or shows no GPU label, it's on the free tier. You can fork it and run your own copy for free.

Five Strategies That Make CPU Work

1. Choose Small, Specialized Models

A 100M-parameter model trained specifically for NER will often outperform a 7B general-purpose model on NER tasks — and run 100x faster on CPU. The GLiNER models on the leaderboard demonstrate this. Small and focused beats large and general.

2. ONNX Runtime

ONNX (Open Neural Network Exchange) optimizes models for inference. Converting a PyTorch model to ONNX and running it with onnxruntime can give 2–5x speedups on CPU. The runtime is also much smaller than full PyTorch, which means faster startup.

3. Quantization

Quantization reduces model precision from 32-bit floats to 8-bit integers (or even 4-bit). This shrinks the model by 4–8x and speeds up CPU inference significantly, usually with minimal quality loss. Libraries like optimum make this straightforward.

4. API Delegation

Some of the most popular CPU Spaces don't run models locally at all. They call external APIs. Edge TTS calls Microsoft's speech API. Gradio Playground Bot calls an LLM API. The Space is just a frontend — all the heavy compute happens elsewhere.

This is such an important pattern that it has its own chapter: Chapter 4: Free APIs.

Tradeoff: API delegation means your Space depends on an external service. If the API goes down or changes terms, your Space breaks. Free APIs may have rate limits. But for a student project, this is often the right call.

5. No Inference at All

The Open ASR Leaderboard and Big Code Models Leaderboard are among the most-liked Spaces on Hugging Face, and they run zero ML inference. They're pure data visualization: tables, charts, and filters on pre-existing benchmark data. You can build a hugely popular Space with nothing but Gradio's data display components and a CSV file.

Red Flags: When a Space Probably Needs GPU

When you're browsing Spaces and wondering whether something could run on CPU, watch for:

  • Diffusion models (Stable Diffusion, FLUX) — Image generation almost always needs GPU.
  • Video generation — Frame-by-frame inference is extremely compute-heavy.
  • Large language models (>3B params) — Chatting with Llama 7B or Qwen 7B requires GPU.
  • Real-time inference on large models — If it responds in under a second with a large model, it's on GPU.
  • GPU-specific packages in requirements.txt — xformers, flash-attn, or CUDA-specific bitsandbytes.

The Sweet Spot

The best CPU-only Spaces live in a sweet spot: they use small, specialized models for focused tasks, or they avoid live inference entirely by using APIs, precomputation, or pure data visualization. They don't compete with GPU Spaces on raw model power. They compete on usefulness, design, and clever architecture.
