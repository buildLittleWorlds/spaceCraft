Method 7 · Session 8

Error Propagation

Break step 1 on purpose. Then watch the damage cascade through every step that follows.

What It Is

Error propagation means tracing how a mistake in one part of a system causes mistakes in every part after it. In a multi-model pipeline, the output of Model 1 becomes the input of Model 2. If Model 1 is wrong, Model 2 has no way to know — it just processes whatever it receives. The error cascades forward, often getting worse at each step.

This is how real production systems fail. A self-driving car's camera misreads a sign, the planning module makes a bad decision based on the misread, and the car turns the wrong way. Each component worked correctly on its input — but the input was corrupted. Understanding error propagation is how you build systems that fail gracefully instead of catastrophically.

When You Use It

Use error propagation testing whenever a system has more than one step — any time one model's output feeds into another model, API, or decision. This includes any Space that chains models together, combines an API with a classifier, or processes data through multiple stages.

The Steps

  1. Map the chain. Draw every step: what goes in, what comes out, which model or API sits in between. For example:
    Image → Caption model → Caption text → Sentiment model → Mood label
  2. Break step 1 on purpose. Feed the first model an input you know will cause trouble — an image it can't caption well, text in a language it doesn't know, an edge case from adversarial testing.
  3. Trace the error forward. Feed that flawed output into step 2 without correcting it. Then into step 3. At each stage, record:
    • Does the error get worse, stay the same, or accidentally get corrected?
    • Can you tell from the final output that step 1 was the problem?
    Bad image → Wrong caption → Wrong mood
  4. Find the fragile points. Which connections in the chain are most vulnerable? Where does one bad output cause the most downstream damage?
  5. Trace backward from the final output. When you see a wrong final result, resist the urge to fix the last step. Instead, trace backward: was the input to the last step already wrong? Was the input to the step before that wrong? The bug is usually upstream.

Connection to Chapter 7: The Multi-Model Pipelines chapter covers how to build pipelines with error handling. This method is the testing companion — how to find where your pipeline is fragile so you can add validation between steps.

See It on SpaceCraft

CLIP Interrogator

What to try: CLIP Interrogator chains image embedding extraction with text generation to describe an image. Upload a clear, well-lit photo — then upload something ambiguous: a blurry image, an abstract painting, a photo with misleading context. How does the quality of the generated description change? When the image embedding step struggles, the text generation step inherits that confusion.

Pathfinder

What to try: Pathfinder retrieves scientific papers by converting your query into an embedding, then finding similar embeddings in its database. Try a clear, specific query ("transformer attention mechanisms") and then a vague or misleading one ("things that are kind of related to that one paper"). When the query embedding is poor, the retrieval results degrade — but they still look like confident results. The pipeline doesn't tell you it's uncertain.

What You'll Discover

The key insight is that a pipeline is only as reliable as its weakest link, and the weakest link is usually the connection between steps, not the steps themselves. Each model in a chain may work perfectly on clean input. The failure happens at the handoff — where one model's output becomes the next model's input without any validation. The fix isn't better models. The fix is checking the intermediate results and building in validation between steps, as covered in Chapter 7.

Try it yourself: Think of any two Spaces on the leaderboard whose outputs and inputs could connect — for example, an OCR Space that produces text, followed by a sentiment analysis Space that reads text. Mentally chain them: what would happen if the OCR output was garbled? Would the sentiment classifier know? Write down where you'd add a validation check between the two steps.
