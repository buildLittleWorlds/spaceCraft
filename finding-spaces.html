Finding Spaces to Study

The fastest way to get better at building Spaces is to read other people's code. Here's how to find the good ones.

Where to Browse

The main directory is at huggingface.co/spaces. By default it shows trending Spaces, which tend to be GPU-heavy. You need to look deeper.

Sort ByWhat You'll FindCPU Gems
TrendingWhat's popular right nowOccasionally a lightweight tool goes viral
Most LikesAll-time popularLeaderboards, dashboards, classic tools
Most RecentNewest SpacesMany start on CPU before upgrading

Best strategy: Sort by "Most Likes" and scroll. The all-time popular list includes many CPU-only tools mixed in with GPU generators. These long-lived CPU Spaces are the best to study because they've been refined over time.

The search bar searches across names, descriptions, and tags:

How to Confirm a Space Runs on CPU

When you find something interesting:

  1. Check the hardware badge on the Space page. "CPU Basic" or no GPU label = free tier.
  2. Click "Files" and check the README.md frontmatter. No hardware: field means free CPU.
  3. Read app.py. Small models, pipeline-only inference, and API calls all mean CPU-friendly.
  4. Check requirements.txt. No torch at all (just gradio and an API client) = almost certainly CPU. Has torch but no xformers or flash-attn = probably CPU.
  5. Fork it. The ultimate test. Fork to your free account and see if it runs.

Reading a Space's Code

Once you find a CPU Space worth studying, here's what to look for in the source:

Model Loading Patterns

  • pipeline("task", model="...") — HF pipeline. Check the model name: small models (under 500M params) work on CPU. (Chapter 3)
  • onnxruntime or .onnx files — Optimized for CPU inference. (Chapter 3)
  • requests.post() or API client library — Delegating compute externally. (Chapter 4)
  • No model at all, just CSV/JSON data — Pure visualization. (Chapter 3)

Gradio Components

The components tell you what kind of interface it builds:

  • gr.Textbox + gr.Textbox — Text in, text out. Classic NLP tool.
  • gr.File + gr.Textbox — Document upload and extraction (OCR).
  • gr.Dataframe — Data display. Leaderboards and dashboards.
  • gr.Audio — Audio input or output. TTS and ASR tools.
  • gr.Dropdown + gr.Textbox — Model or option selection.

Community Signals

How to judge quality before diving into code:

  • Likes — A CPU Space with 100+ likes is almost certainly worth studying.
  • Recent commits — Check the Files tab. Recent updates mean it likely works with current libraries.
  • Discussions — The Discussions tab shows user feedback. Are people using it successfully?
  • Forks — A highly-forked Space is one other builders found worth copying. Strong signal.

Four Patterns Worth Studying

After reviewing dozens of CPU Spaces for the leaderboard, four architectural patterns stood out. Each one is a template you can build on.

1. The API Wrapper

Build a polished Gradio interface that calls a free API for the heavy lifting. Your Space handles UI, input validation, and output formatting. All compute happens elsewhere.

Leaderboard examples: Edge TTS, Gradio Playground Bot, Qwen3 Demo

2. The Specialized Model

Use a small model trained for one task. It won't win general benchmarks, but it'll be fast and effective in its niche.

Leaderboard examples: GLiNER HandyLab, BRIA RMBG 1.4, AI Content Detector

3. The Data Dashboard

No ML inference at all. Load pre-computed data and let users explore it with filters, sorts, and visualizations.

Beyond the leaderboard: Spaces like the Open ASR Leaderboard (1,200+ likes) show how pure data visualization can be wildly popular with no ML inference at all.

4. The Crowd Tool

Let users contribute data rather than consuming compute. Voting systems, annotation tools, and submission platforms are inherently CPU-friendly because the "work" is done by the users.

Beyond the leaderboard: The TTS Spaces Arena lets users vote on which voice sounds better — all the "work" is done by human evaluators, not compute.

Exercise: Find a CPU-only Space not on the SpaceCraft leaderboard. Fork it. Read its code. Write a short analysis: what model or API does it use? How does it handle user input? What makes it work on CPU? What would you change? Then build your own variation. That cycle — find, study, build, improve — is how you get good at this.
