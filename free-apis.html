Free APIs

The most powerful CPU strategy: let someone else's servers do the heavy lifting. Your Space just handles the interface.

The Pattern

Some of the most popular CPU-only Spaces on the leaderboard don't run ML models at all. They call an external API that does the work, then display the results in a polished Gradio interface. Your Space is the frontend. The API is the backend.

This is not cheating. This is how professional software is built. Almost every app you use daily works this way: your phone is a thin client calling APIs that run on powerful servers somewhere else.

How It Works in a Space

The basic flow:

  1. User enters input in your Gradio interface
  2. Your app.py sends a request to an external API
  3. The API processes it (on their hardware, with their GPU if needed)
  4. Your Space receives the result and displays it

Your Space uses almost zero compute. All the heavy work happens on the API provider's servers.

A Simple Example

Here's what an API-powered Space looks like. This calls a news API and displays results:

import gradio as gr import requests import os def get_headlines(topic): api_key = os.environ.get("NEWS_API_KEY") url = "https://gnews.io/api/v4/search" params = { "q": topic, "lang": "en", "max": 5, "apikey": api_key } response = requests.get(url, params=params) data = response.json() if "articles" not in data: return "No results found." results = [] for article in data["articles"]: results.append( f"**{article['title']}**\n" f"{article['description']}\n" f"[Read more]({article['url']})\n" ) return "\n---\n".join(results) demo = gr.Interface( fn=get_headlines, inputs=gr.Textbox(label="Search topic"), outputs=gr.Markdown(label="Headlines"), title="World Headlines", description="Search for news on any topic." ) demo.launch()

That's a complete Space. It uses the GNews API (free tier: 100 requests/day) and needs no ML model at all. The requirements.txt is just two lines: gradio and requests.

Key detail: The API key is stored as a Secret in your Space settings (see Chapter 2), not in the code. os.environ.get("NEWS_API_KEY") reads it from the environment at runtime.

The Hugging Face Inference API

This one deserves special attention. Hugging Face itself provides a free Inference API that lets you run any public model on their servers. You send a request with your input, they run the model on their hardware (often GPU), and send the result back.

import requests import os API_URL = "https://api-inference.huggingface.co/models/facebook/bart-large-mnli" headers = {"Authorization": f"Bearer {os.environ.get('HF_TOKEN')}"} def classify(text, labels): payload = { "inputs": text, "parameters": {"candidate_labels": labels.split(",")} } response = requests.post(API_URL, headers=headers, json=payload) return response.json()

This means you can build a Space that uses a large zero-shot classification model — running on HF's GPUs — from a free CPU Space. Your Space is just the interface. The model runs elsewhere.

Rate limits: The free HF Inference API is rate-limited and may queue your request if the model isn't loaded. For a student project or demo, this is fine. For a production tool used by hundreds of people, you'd need a paid plan or a different approach.

The API-Wrapper Architecture

When you build an API-powered Space, your code has a clear structure:

  1. Input handling — Gradio collects user input (text, files, selections)
  2. API call — Your function sends a request to the external service
  3. Response parsing — You extract the useful data from the API response
  4. Output formatting — You present the result in a clean, readable way

Your value as a builder isn't in running the model. It's in choosing the right API, designing the interface, handling errors gracefully, and presenting results clearly. That's product design, and it's a real skill.

Combining APIs

The most interesting API-powered Spaces combine multiple APIs. For example:

  • A news analyzer that fetches headlines from GNews, then classifies their sentiment using the HF Inference API
  • A travel tool that looks up a country via REST Countries and weather via Open-Meteo, then summarizes the result
  • An image captioner that fetches a photo from Unsplash and describes it using the HF Inference API with a vision model

Each of these uses zero local compute. All the ML runs on HF's Inference API. All the data comes from free APIs. Your Space is the glue.

Managing API Keys

Some APIs require a key (usually free to get). Here's the workflow:

  1. Sign up for the API and get your key
  2. Add it as a Secret in your Space settings on Hugging Face (Settings → Variables and secrets → New secret)
  3. Read it in your code with os.environ.get("SECRET_NAME")

Never put API keys in your source code. The git history is public.

What to Watch Out For

  • Rate limits — Every free API has limits. Check the docs and design your Space to stay within them. Show a friendly message if the limit is hit.
  • API changes — Free APIs can change their terms, rate limits, or response format at any time. Your Space might break if the API changes.
  • Latency — API calls add network round-trip time. A local model gives instant results; an API call might take 1-3 seconds. Set user expectations with loading indicators.
  • Error handling — APIs can be down, return errors, or return unexpected data. Always wrap API calls in try/except and give users a clear message when something goes wrong.

Design principle: An API-powered Space should feel like a local tool. The user shouldn't have to think about where the computation is happening. If the API is slow, show a loading state. If it errors, show a helpful message. If the rate limit is hit, tell the user to try again later. Hide the plumbing, show the polish.

Spaces on the Leaderboard That Use This Pattern

  • Edge TTS — Microsoft's speech API via the edge-tts library. No key needed.
  • Text-To-Speech Unlimited — API-based TTS with no character limits.
  • Gradio Playground Bot — Uses an LLM API to generate Gradio app code.

Study their code. See how they handle API calls, format responses, and manage errors. Then build your own.
