Chapter 7

Multi-Model Pipelines

What happens when one model's output becomes the next model's input? You get a pipeline. You also get error cascades.

What's a Pipeline?

A pipeline chains two or more steps together. The output of step 1 becomes the input to step 2. Each step can be a model, an API call, or a data transformation.

User Input → Step 1: Get Data → Step 2: Analyze → Output

You've already seen single-step Spaces: user types text, model classifies it, result appears. A pipeline does that same thing, but with multiple steps in sequence. And every step on free CPU.

Example 1: API + Local Model

This pipeline fetches news headlines (API call), then classifies the sentiment of each one (local model). Two steps, zero GPU.

import gradio as gr import requests import os from transformers import pipeline # Step 2: Local sentiment model (67M params, runs on CPU) classifier = pipeline( "sentiment-analysis", model="distilbert-base-uncased-finetuned-sst-2-english" ) def analyze_headlines(topic): # Step 1: Fetch headlines from GNews API api_key = os.environ.get("GNEWS_API_KEY") url = "https://gnews.io/api/v4/search" params = {"q": topic, "lang": "en", "max": 5, "apikey": api_key} try: response = requests.get(url, params=params, timeout=10) data = response.json() except Exception: return "Could not fetch headlines. Try again." if "articles" not in data or not data["articles"]: return f"No headlines found for '{topic}'." # Step 2: Classify each headline results = [] for article in data["articles"]: title = article["title"] sentiment = classifier(title)[0] label = sentiment["label"] score = round(sentiment["score"] * 100) results.append(f"**{label}** ({score}%) — {title}") return "\n\n".join(results) demo = gr.Interface( fn=analyze_headlines, inputs=gr.Textbox(label="News topic", placeholder="e.g. climate change"), outputs=gr.Markdown(label="Headlines with sentiment"), title="News Sentiment Pipeline", description="Fetches headlines via GNews, then classifies " "each one's sentiment using DistilBERT on CPU." ) demo.launch()

What's happening: Step 1 (GNews API) runs on GNews's servers. Step 2 (DistilBERT) runs locally on your free CPU. The pipeline connects them: headlines flow from the API into the classifier. Neither step alone is remarkable. Together, they're a useful tool.

Example 2: Chaining Two HF Inference API Calls

If both steps are too heavy for local CPU, you can chain two remote API calls using the HF Inference API:

User uploads image → API: Caption image → API: Classify caption → Image mood
import gradio as gr import requests import os HF_TOKEN = os.environ.get("HF_TOKEN") headers = {"Authorization": f"Bearer {HF_TOKEN}"} def caption_image(image_path): """Step 1: Generate a caption for the image.""" with open(image_path, "rb") as f: response = requests.post( "https://api-inference.huggingface.co/models/Salesforce/blip-image-captioning-base", headers=headers, data=f.read() ) result = response.json() if isinstance(result, list) and len(result) > 0: return result[0].get("generated_text", "") return "" def classify_sentiment(text): """Step 2: Classify the caption's sentiment.""" response = requests.post( "https://api-inference.huggingface.co/models/distilbert-base-uncased-finetuned-sst-2-english", headers=headers, json={"inputs": text} ) result = response.json() if isinstance(result, list) and len(result) > 0: top = result[0][0] return f"{top['label']} ({round(top['score']*100)}%)" return "Unknown" def image_mood(image): caption = caption_image(image) if not caption: return "Could not generate a caption.", "" mood = classify_sentiment(caption) return caption, mood demo = gr.Interface( fn=image_mood, inputs=gr.Image(type="filepath", label="Upload an image"), outputs=[ gr.Textbox(label="Caption (Step 1)"), gr.Textbox(label="Mood (Step 2)") ], title="Image Mood Pipeline", description="Captions the image (BLIP), then classifies " "the caption's sentiment (DistilBERT). Both via API." ) demo.launch()

Your Space has zero model loading. Both BLIP and DistilBERT run on Hugging Face's servers. Your Space is just the orchestration layer.

Error Cascades

Here's the thing about pipelines: errors propagate. If Step 1 produces bad output, Step 2 processes bad input. The final result is wrong, and it's not obvious where the error started.

Example: If the image captioning model says "a cat sitting on a chair" but the image is actually a dog, the sentiment classifier will correctly classify the caption's sentiment — but the whole result is wrong, because the caption was wrong. The classifier has no way to know the caption was bad. It just processes what it gets.

This is why pipelines are harder to debug than single-step tools. When something goes wrong, you need to check each step independently:

  1. Is Step 1's output reasonable? Print or display it separately.
  2. Is Step 2 getting good input? Show the intermediate result to the user.
  3. Is the error in the data, the model, or the connection between them?

Defensive Pipeline Design

Build pipelines that handle failure gracefully:

Show Intermediate Results

Don't just show the final output. Show what each step produced. In the image mood example above, we return both the caption and the mood as separate outputs. If the caption is wrong, the user can see why the mood is off.

Validate Between Steps

# After Step 1 caption = caption_image(image) if not caption or len(caption) < 5: return "Caption too short or empty — skipping analysis.", "" # Only proceed to Step 2 if Step 1 succeeded mood = classify_sentiment(caption)

Wrap Each Step in Error Handling

try: caption = caption_image(image) except Exception as e: return f"Step 1 failed: {e}", "" try: mood = classify_sentiment(caption) except Exception as e: return caption, f"Step 2 failed: {e}"

Pipeline Patterns on the Leaderboard

Several Spaces on the SpaceCraft leaderboard use pipeline-like architectures:

  • GLiNER HandyLab — Runs entity extraction (step 1), then uses those entities for downstream QA and summarization (step 2)
  • Summarize Long Text — Chunks long documents (step 1), then summarizes each chunk and combines (step 2)
  • Face Swap — Detects faces (step 1), then applies the swap transformation (step 2)

When to Use a Pipeline vs. a Single Model

  • Use a single model when one model does the whole job well enough. Simpler is better.
  • Use a pipeline when no single model does what you need, but two models combined can. For example: no model translates AND classifies sentiment, but you can chain a translator and a classifier.
  • Use an API pipeline when the models you need are too large for CPU. Let the HF Inference API run them remotely and chain the results in your Space.

The key insight: A pipeline is only as reliable as its weakest step. Every additional step is a place where things can go wrong. Build the simplest pipeline that solves your problem, show intermediate results so users can verify, and handle errors at every step.
