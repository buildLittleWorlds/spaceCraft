Anatomy of a Hugging Face Space

A Space is a git repo that Hugging Face turns into a live web app. Here's what's inside.

The Core Idea

A Hugging Face Space is a git repository that Hugging Face automatically builds and deploys as a web application. You push code, HF builds a Docker container from it, and your app goes live at huggingface.co/spaces/your-username/your-space.

No server configuration. No Docker knowledge required. No domain setup. You push, it deploys.

Choosing an SDK

When you create a Space, you pick an SDK. This determines what framework runs your app.

SDKLanguageBest For
GradioPythonML demos, model interfaces, data tools
StreamlitPythonData dashboards, interactive reports
StaticHTML/JS/CSSLanding pages, client-side apps, docs
DockerAnyCustom setups, non-Python apps

Start with Gradio. It's the most common SDK on Hugging Face, it has the best documentation, and nearly every Space on the leaderboard uses it. Gradio gives you pre-built UI components for file uploads, text inputs, sliders, audio players, image displays, and more.

Gradio vs Streamlit

Both turn Python scripts into web apps. Gradio is designed for ML interfaces (input goes in, model output comes out). Streamlit is designed for data dashboards (explore and visualize data). Most HF Spaces use Gradio because most Spaces are model demos.

When to Use Static

If your Space doesn't need a Python backend — like a leaderboard that displays pre-computed data, or a documentation site — use the Static SDK. It serves plain HTML/CSS/JS with no server process, starts instantly, and uses zero compute. (SpaceCraft itself could be a Static Space.)

The App File

Every Gradio or Streamlit Space needs an entry point:

  • Gradio / Streamlit: app.py
  • Static: index.html

A minimal Gradio Space:

import gradio as gr def greet(name): return f"Hello, {name}!" demo = gr.Interface(fn=greet, inputs="text", outputs="text") demo.launch()

That's a complete, deployable Space. Push it to HF and it goes live. Chapter 5 walks through building something like this step by step.

The Configuration File

Every Space has a README.md with YAML frontmatter that configures it:

--- title: My Space emoji: 🔬 colorFrom: blue colorTo: green sdk: gradio sdk_version: 5.0.0 app_file: app.py pinned: false ---

This tells Hugging Face which SDK to use, what version, and which file to run. The emoji, colorFrom, and colorTo control how your Space looks in search results.

Dependencies

If your Space needs Python packages beyond Gradio, list them in requirements.txt:

transformers torch sentencepiece

CPU tip: Package size matters. PyTorch alone is ~800 MB. If you only need inference, consider onnxruntime instead (much smaller). Or if you're using a free API, you may not need a model framework at all — just requests and gradio.

Secrets and Environment Variables

If your Space calls an external API (see Chapter 4), you'll likely need API keys. Store them as Secrets in your Space settings — they're injected as environment variables at runtime.

import os api_key = os.environ.get("MY_API_KEY")

Never put API keys directly in your code. Secrets are encrypted and hidden from the git history.

Persistent Storage

By default, Space storage is ephemeral — anything written to disk disappears when the Space restarts. Most CPU Spaces on the leaderboard don't need persistent storage. They process inputs in real-time and return results immediately.

If you're building a leaderboard or dashboard, the data is usually stored in the git repo itself (as a CSV or JSON file) and updated via commits.

The Minimal CPU Space

Here's what the simplest useful CPU Space looks like:

my-space/ ├── README.md # YAML config (sdk, title, emoji) ├── app.py # Your Gradio or Streamlit app └── requirements.txt # Python dependencies (if any)

Three files. That's all you need to have a live, public, shareable application. The question isn't whether you have enough resources. The question is: what will you build?

Before you answer that, you should understand what "CPU" and "GPU" actually mean for your Space. That's Chapter 3.
