Chapter 8

Designing for Your Audience

The same model can be a confusing demo or a useful tool. The difference is design.

Why Design Matters on Free CPU

When you're building on free CPU, you can't compete on raw model power. You can't run the biggest model or generate the fastest results. What you can compete on is how your Space feels to use. A well-designed Space with a small model will always beat a poorly-designed Space with a powerful model.

On the SpaceCraft leaderboard, every Space is rated on "Craft" — is the interface well-designed? The highest Craft scores don't go to the most technically complex Spaces. They go to the ones where someone clearly thought about the person using it.

Pick One Person

The most common design mistake in student Spaces is building for "everyone." A sentiment analysis tool for everyone is a generic demo. A sentiment analysis tool for a restaurant owner reading reviews is a product.

Before you write any code, answer these questions:

  1. Who is this for? One specific type of person. Not "anyone." A teacher. A journalist. A student studying for exams. A podcaster.
  2. What are they trying to do? What task do they need to complete? What question do they need answered?
  3. What do they already know? A machine learning researcher knows what "sentiment" means. A restaurant owner doesn't care about the word "sentiment" — they care about whether a review is positive or negative.

The test: Can you describe your Space in one sentence that your target user would understand? Not "This Space uses DistilBERT for binary sentiment classification." Instead: "Paste a restaurant review and see if it's positive or negative."

Titles and Descriptions

Your title and description are the first things anyone sees. They determine whether someone tries your Space or moves on.

Weak

Title: "Sentiment Analysis Demo"

Description: "This Space uses the distilbert-base-uncased-finetuned-sst-2-english model for sentiment classification."

Strong

Title: "Review Checker"

Description: "Paste a customer review and see if it's positive or negative. Built for small business owners who want to quickly scan feedback."

The weak version tells you what the model is. The strong version tells you what the Space does for you. In code:

demo = gr.Interface( fn=analyze, inputs=gr.Textbox( label="Paste a review", placeholder="e.g. The food was amazing but the service was slow...", lines=3 ), outputs=gr.Markdown(label="Verdict"), title="Review Checker", description="Paste a customer review and see if it's positive " "or negative. Built for small business owners." )

Examples That Teach

Gradio lets you add example inputs that users can click to try instantly. Good examples do three things:

  1. Show the Space works — The user clicks, sees a result, and understands the tool immediately.
  2. Demonstrate the range — Include at least one positive case, one negative case, and one ambiguous case.
  3. Match the audience — If your Space is for restaurant owners, use restaurant reviews as examples. Not movie reviews, not tweets.
demo = gr.Interface( fn=analyze, inputs=gr.Textbox(label="Paste a review", lines=3), outputs=gr.Markdown(label="Verdict"), title="Review Checker", examples=[ ["The pasta was incredible and the staff was so friendly!"], ["Waited 45 minutes for cold food. Never coming back."], ["It was fine. Nothing special but nothing wrong either."], ] )

Output Formatting

Raw model output is not human output. The model returns a JSON object with labels and scores. Your user wants a clear answer.

Raw output

[{"label": "POSITIVE", "score": 0.9987}]

Formatted output

POSITIVE (99.9% confidence)
This review is clearly positive. The language suggests strong satisfaction.

Format your output for the person using it:

def format_result(result): label = result["label"] score = round(result["score"] * 100, 1) if label == "POSITIVE": emoji = "✅" interpretation = "This review is positive." else: emoji = "⚠️" interpretation = "This review is negative." if score > 95: confidence = "Very confident" elif score > 80: confidence = "Fairly confident" else: confidence = "Uncertain" return f"{emoji} **{label}** ({score}%)\n\n{confidence}. {interpretation}"

Error States

When something goes wrong, your user should know what happened and what to do about it. Never show a Python traceback to a user.

Bad error

IndexError: list index out of range

Good error

Could not analyze this text. It may be too short or in an unsupported language. Try a longer English sentence.

def analyze(text): if not text or not text.strip(): return "Please enter some text to analyze." if len(text.strip()) < 10: return "Text is too short for reliable analysis. Try a full sentence." try: result = classifier(text)[0] return format_result(result) except Exception: return ("Something went wrong. This can happen with very " "unusual text. Try a different input.")

Loading States

On free CPU, models can take a few seconds to respond. If your Space calls an API (Chapter 4) or runs a pipeline (Chapter 7), the delay may be longer. Gradio shows a loading spinner automatically, but you can improve the experience:

  • Set expectations in the description: "Analysis takes 2-3 seconds on free CPU."
  • First load is slowest: When your Space wakes from sleep (after 48h), the first request loads the model into memory. Warn users: "If the Space was sleeping, first request may take 30 seconds."
  • For pipelines, show progress: Return intermediate results so the user sees something happening at each step.

Accessibility Basics

Good design means your Space works for everyone who might find it:

  • Label your inputs. gr.Textbox(label="Paste a review") is better than gr.Textbox(). Screen readers use labels.
  • Don't rely on color alone. If you use green/red to show positive/negative, also include the word "POSITIVE" or "NEGATIVE." Some users are colorblind.
  • Write clear placeholder text. placeholder="e.g. The food was amazing..." tells users exactly what to type.
  • Keep the interface simple. One input, one output, clear labels. Don't add controls you don't need. Every slider and dropdown is a decision the user has to make.

The Craft Checklist

Before you share your Space, run through this list. These are the things that separate a demo from a tool:

  • Does the title say what the Space does (not what model it uses)?
  • Does the description name the audience?
  • Are there 2-3 example inputs the user can click?
  • Is the output formatted for a human (not raw JSON)?
  • Does it handle empty input gracefully?
  • Does it handle errors without showing tracebacks?
  • Are all inputs labeled?
  • Would someone who doesn't know ML understand what to do?

Try this: Share your Space URL with someone who wasn't involved in building it. Don't explain anything. Watch what they do. Do they understand the title? Do they know what to type? Do they understand the output? Every point of confusion is a design opportunity.

Design on the Leaderboard

  • Descriptions that tell you who it's for
  • Example inputs you can click

  • Graceful handling of edge cases
